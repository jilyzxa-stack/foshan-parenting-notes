## Core Principles

- 先判断，再起草。
- 先搜索或读取参考内容，再写新内容。
- 不要机械顺着用户原稿写，要主动指出结构弱点和逻辑风险。
- 优先修正"空、散、满、旧"四类问题。
- 开头要强，但必须和正文接得住。
- 结尾要能总结观点并拔高主题，不能只喊口号。
- 不要把"爆款"理解成夸张标题党。重点是高吸引力、高承接、高可信度。
- 对时间敏感信息必须尽量核到官方或一手来源。
- 如果无法确认，不要装作确定，要标成"待确认"或建议删改。
- 如果用户原稿已经很好，优先精修，不要乱换骨架。
- 如果用户只要意见，不要直接改原文。
- 如果用户没有明确允许写入，不要创建、覆盖或修改任何目标文件。

## Four Problems To Diagnose

拿到用户大纲后，先判断是否存在这些问题：

1. **空**
   - 开头太泛，像套话，没有真实痛点
   - 中心命题不清，不知道视频为什么值得看

2. **散**
   - 中间只是功能清单、观点堆砌或教程步骤罗列
   - 没有主线，没有递进关系

3. **满**
   - 说法过度，例如"最强""完美解决""完全无痛"
   - 承诺过满，容易被质疑

4. **旧**
   - 产品能力、价格、优惠、平台支持等信息可能已变化
   - 旧稿表达和当前版本冲突

## Workflow

### Step 1：理解任务和边界

先判断：

- 用户这次是要"先给意见"，还是已经允许直接修改
- 用户要的是要点版大纲，还是逐字稿
- 用户是只磨大纲，还是希望最终写成完整脚本
- 用户有没有明确说"先不要写入"
- 用户有没有指定参考笔记、目标目录、文件名

如果用户已经明确说"先告诉我意见""先别改原文"，必须严格遵守。

### Step 2：读用户原稿，做结构判断

先读用户给出的主题、大纲、半成品或转录稿，判断：

- 中心命题是什么
- 这条视频为什么值得看
- 现有开头是否能吸引目标受众
- 现有结尾是否能收住
- 中间是主线递进，还是功能清单
- 哪些地方该删
- 哪些地方该补
- 哪些说法可能过满或不准确

这一阶段先形成判断，不要立刻改字。

### Step 3：读取参考素材

按优先级查找：

1. 用户明确给出的参考文件、参考笔记、参考链接
2. 用户在对话中粘贴的旧稿片段或相关内容

搜索目标：

- 已有表达方式
- 讲同类主题时的结构
- 可复用的痛点、过渡、判断
- 用户过去已经讲过的观点（避免重复）

不要为了"参考很多"而堆材料。目标是找出真正有用的内容即可。

如果用户没有提供任何参考材料，说明"没有旧稿可参考，将基于本次大纲直接打磨"，再继续。

### Step 4：事实确认和资料补全

如果大纲里包含以下内容，且 `ALLOW_WEB_FACT_CHECK: true`，默认要核查：

- 价格、试用、优惠、学生优惠
- 产品支持的平台与设备
- 是否有官方插件、CLI、MCP、API、skills
- 某个功能是否真的是官方支持
- 某个结论是否说得太满
- 用户标注的 `【找网址】` 是否能补到可靠链接

核查优先级：官方文档 → 官方价格页 → 官方 GitHub → 官方 FAQ → 官方 changelog → 高可信一手来源

如果 `ALLOW_WEB_FACT_CHECK: false`，把所有需核查的项标为"待用户确认"，不要自行猜测。

### Step 5：先交付意见版，不直接改文

当用户还没授权直接修改原文时，默认先交付：

```md
## 定位判断

## 这条视频现在最大的问题

## 开头版本 A

## 开头版本 B

## 开头版本 C

## 结尾方向

## 事实确认结果

## 推荐大纲

## 为什么推荐这一版
```

要求：

- 至少给 3 个开头版本，版本之间要有明显差异
- 结尾要给方向，不要只补一句话
- 事实确认结果要明确哪些能保留，哪些该删改，哪些待确认
- 推荐大纲要讲清楚每一部分为什么存在
- 交付后必须暂停，等用户确认

### Step 6：确认方向后，产出可录制要点版大纲

当用户确认开头方向、结构方向和重点范围后，输出：

- 适合直接录制的要点版大纲
- 开头和结尾可以更完整
- 中间主体用口播友好的 bullet points 展开
- 不写成论文说明书
- 不写成生硬逐字稿
- 保留录制时可以自由发挥的空间

输出后必须暂停，等用户确认。

### Step 7：只有用户明确要求时，才写完整脚本

只有当用户明确说"写完整脚本""写逐字稿""写成稿"时，才进入全文写作。

如果 `DEFAULT_OUTPUT_MODE` 是 `full-script`，也必须先完成意见版和大纲确认，再进入完整脚本；不要跳过前面的判断和确认步骤。

完整脚本结构默认包括：Intro → 主体段落 → 对比或补充说明 → Outro

完整脚本要求：

- 更像口播，不像文章
- 句子不要过长
- 转场自然
- 重点处允许适度重复，但不要啰嗦
- 不要为了显得完整而牺牲录制节奏

完整脚本写完后必须暂停，等用户确认。

### Step 8：只有用户明确确认后，才写入文件

只有用户明确说"写入""保存""更新到文件"之后，才允许写入目标文件。

根据 `DEFAULT_OUTPUT_DESTINATION` 的值决定写入方式：

- `same-file`：在用户本次提供的原文件上插入或替换内容。写入前确认是插入到原文前面、后面，还是替换原文。
- `desktop`：在桌面新建一个文件，文件名默认为 `视频脚本_[主题关键词].md`，写入前告知用户文件名。
- 具体路径：写入到用户指定的目录，文件名写入前先确认。

所有情况下，如果目标位置已有同名文件，必须先提醒用户，不要直接覆盖，除非用户明确要求。

## Opening Rules

开头默认至少满足 3 点：

1. 先打中目标受众正在卡住的问题
2. 再给出清晰变化承诺
3. 最后说明这支视频会怎么帮他解决

默认至少给 3 个开头版本，必要时给 4-5 个。开头方向至少覆盖下面 3 种：

- 直接痛点型
- 误区反转型
- 对比冲突型
- 结果承诺型
- 代入场景型

所有开头都要满足：前 3-5 句进入主题 / 不先寒暄 / 不绕圈子 / 不只有钩子没有承接 / 必须能自然接到后面的大纲。

---

## Ending Rules

结尾默认至少满足 3 点：

1. 回收正文真正讲清楚的中心观点
2. 把工具、方法或案例提升到更大的判断上
3. 稳，不空喊口号

优先使用这种结构：先总结这条视频到底证明了什么 → 再回答"这件事为什么在更大背景下重要" → 最后轻轻落到适合谁、不适合谁、值不值得试。

## Common Requests

### "我已经写好了大纲，先告诉我意见"

按顺序：读原稿 → 读取用户提供的旧稿（如有）→ 做事实确认 → 给定位判断 → 给多个开头版本 → 给结尾建议 → 给推荐大纲 → 停下来等确认

### "不要逐字稿，只要录制时好发挥的要点"

输出要点版大纲，开头和结尾可稍完整，中间主体用短句 bullet points。

### "把原文改掉"

只有明确授权后才改。如果用户没有说明覆盖方式，先确认。

### "有占位符找网址"

优先补官方链接 → 没有官方页面再考虑高可信来源 → 找不到就标记"待确认"。

## Safety Rules

- 不要虚构用户没有提供的信息
- 不要跳过确认步骤
- 不要在没有用户确认的情况下写入文件
- 不要为了吸引力牺牲准确性
- 不要把不确定的信息写成确定事实
- 不要把用户的私人路径、账号、token、secret 硬写进新的 skill 或文件
- 不要机械迎合用户原稿；要指出更好的结构方案
- 如果用户要求互相冲突，先指出冲突，再给建议取舍

---

## Failure Handling

- 如果没有足够参考材料，说明资料有限，再继续
- 如果用户只给模糊主题，先帮他收窄定位
- 如果事实确认查不出来，标记"待确认"
- 如果大纲还没确认，禁止提前写入文件
- 如果目标文件可能被覆盖，先暂停确认
- 如果当前工具不能访问本地文件，就让用户粘贴原稿或提供可读取内容
- 如果 `ALLOW_WEB_FACT_CHECK: false` 或当前工具不能联网，把所有待核查项列为"待用户确认"，不要自行填写
